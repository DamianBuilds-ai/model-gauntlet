# STAN_STATE.md - Framework Long-Term Memory

This is the trunk. It is the framework's persistent memory of what it has done,
what is queued, and where results were sent. The `/eval-run` orchestrator reads
it at the start of every run.

---

## How this file works

- **Read at run start.** The first thing `/eval-run` does (after the one-time
  return-channel check) is read this file to learn which evals are done, which are
  queued, and which batch comes next.
- **The orchestrator updates it - never hand-edit mid-run.** After each eval
  completes and is sent, `/eval-run` marks that eval `done` in the Eval queue,
  moves it to Completed, and appends to Run history. Do not edit this file by hand
  while a run is in progress; you will race the orchestrator and corrupt the state.
- **Hand-editing between runs is fine.** Damian (or whoever curates) adds new eval
  specs to the Eval queue and adjusts Next batch between runs. The queue dispatch
  (the second build dispatch) seeds the first real specs here.
- **Resumability.** Because done-state lives here and not in memory, re-running
  `/eval-run` after an interruption skips anything already marked `done` - no
  duplicate work.
